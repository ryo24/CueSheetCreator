# CueSheetCreator — System Architecture & Implementation Policy

本ドキュメントは、将来的に他の開発者やAIアシスタントが本プロジェクトを解析・機能拡張する際のガイドラインとして、企画意図と実装方針をまとめたものです。

## 1. プロジェクトの基本方針設定 (Philosophy)
- **No-Build アーキテクチャ**: イベント現場の運用スタッフ等、IT専門外のユーザーでも「フォルダ内の `index.html` をダブルクリックするだけ」で確実かつ軽量に動作することを最優先としています。Node.js、Webpack、Reactなどの複雑なビルドツールやフレームワークへの依存を意図的に排除し、**純粋な HTML / CSS / Vanilla JavaScript** のみ構成しています。
- **ポータビリティ**: 外部のデータベースやバックエンドサーバーとの通信を必要とせず（Google FontsなどのCDNを除く）、完全なオフライン/ローカルアプリとして機能します。

## 2. ディレクトリ構造
```
CueSheetCreator/
├── index.html        # アプリケーションのエントリーポイント兼メインUI
├── css/
│   ├── style.css     # 画面表示用のメインスタイル（モダンUI/CSS変数を活用）
│   └── print.css     # 現場持ち出し用の印刷（PDF出力）特化レイアウト
├── js/
│   ├── app.js        # 状態管理・DOMレンダリング・入出力のコアロジック
│   ├── jszip.min.js  # オフラインでZIPを処理するための外部ライブラリ
│   └── sortable.min.js # ドラッグ&ドロップでの行並び替え用ライブラリ
└── docs/             # AIおよび開発者向けのドキュメント
```

## 3. データ構造と保存形式 (.cuesheet)
イベントの進行表データと紐づく画像を「1つのファイル」としてポータブルに扱うため、独自の `.cuesheet` 拡張子を採用しています。
- **中身の正体**: `.cuesheet` ファイルの実態はZIPファイルです（`JSZip` を用いてブラウザ上で動的に生成・展開しています）。
- **構造**:
  - `data.json`: テキスト入力内容（イベント名、プログラム構造、シーンごとの進行テキスト等）をJSON化したもの。
  - `images/`: ユーザーが表内にアップロードした画像を格納するディレクトリ。

### JSONデータ構造の具体例 (`data.json` / State Object)
JS内で管理され、`.cuesheet` 内の `data.json` として保存されるコアデータの実際の構造（スキーマ）は以下のようになっています。将来的にAIが別プログラムからこのデータをパース・生成・結合する際の参考にしてください。

## 4. コアロジック (app.js の設計方針)
- **Re-render パターン**: 現代的なフレームワークの動作をVanilla JSで模倣し、データ (`state`) に変更が加わった際は必ず `render()` 関数を呼び出し、必要なDOMを丸ごと再生成・再バインディングするシンプルな単方向データフローを採用しています。
- **画像の自動最適化**: `<canvas>` API を活用して、ユーザーが巨大な画像をD&Dしてもブラウザ内で自動的に幅 800px 以下に圧縮・リサイズし格納(`imageCache`)する仕組みです。
- **オートセーブ**: ブラウザのリロード等でデータが失われないよう、状態が変更されるたびに `localStorage` にJSONデータを書き込み、次回起動時に復元させています。

## 5. 出力（Printing）の最適化
本アプリの最終成果物は「現場の照明・音響・進行スタッフに配付される紙（またはPDF）」です。
- 画面上で見えている「編集用ボタン」や「左サイドバー」等のUIパーツは、`css/print.css` の機能（`display: none !important;` および `.print-only` の制御）によって、印刷ダイアログが起動した瞬間に完全に消去され、現場で使える公式な表組みの形に自動成形されます。
